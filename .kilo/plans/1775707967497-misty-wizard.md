# Plan: Add User Switch Button for Admins in Header

## Goal
Add a "Switch User" button (admin-only) next to the bell icon in the top-bar header. When clicked, a dropdown opens listing all officer names. Selecting an officer filters all CRM data views to show only that officer's data (admin-level impersonation).

## Current State
- **Header location**: `public/index.html` lines 304–355, `.top-bar-right` contains: bell icon → switch-role-wrap → XP widget
- **Existing pattern**: The Switch Role feature (`app.js:4501–4706`, `sidebar.css:479–583`) provides an identical UI pattern — a button with a popup dropdown, populated dynamically, with outside-click-to-close
- **Officers API**: `GET /api/users/officers` (backend `usersRoutes.js:46–101`) returns `{ success, officers: [{ id, name, email }] }`
- **Admin detection**: `window.currentUser.role === 'admin'` and `body.admin` CSS class
- **Data filtering**: Admin endpoints already support `?assignedTo=OfficerName` for filtered views (CRM leads, registrations, dashboard stats). The key is `officer.name` matching the `assigned_to` column.

## How Data Filtering Currently Works

| Module | Admin endpoint | Filter mechanism |
|--------|---------------|-----------------|
| CRM Leads | `GET /api/crm-leads/admin?assignedTo=Name` | Server-side `.eq('assigned_to', name)` |
| Lead meta (batches/sheets) | `GET /api/crm-leads/admin/meta/batches?assignedTo=Name` | Server-side filter |
| Registrations | `GET /api/registrations/admin` | Currently no `assignedTo` filter param |
| Dashboard stats | `GET /api/dashboard/stats` | Server adds per-officer breakdown for admins |
| Calendar tasks | `GET /api/calendar/tasks?mode=officer&officer=Name` | Server filters by owner name |
| Follow-ups | `GET /api/crm-leads/followups/admin/:officerUserId/...` | Uses officer UUID in path |

**Key insight**: Many admin endpoints already accept `assignedTo` as a query param. The frontend just needs to start passing it when an officer is selected.

- `fa-user-secret` icon distinguishes from the role-switch icon (`fa-exchange-alt`)
- Search input for quick filtering when there are many officers

### Step 3: Add JavaScript in `public/js/app.js`

Add a new section **before the Switch Role feature** (~line 4500):

#### `initUserSwitchBtn()` function
- Guard: only for admin users (`window.currentUser.role === 'admin'`)
- Guard: `__userSwitchInitialised` flag to prevent double-init
- Fetch officers via `fetch('/api/users/officers')` with auth header
- Cache officer list in module-level `let _cachedOfficers = []`
- Populate dropdown with: first item = "Admin View (All Data)" reset, then officer items (name + email subtitle)
- Wire up: button click toggles popup, outside-click closes, search input filters list
- Restore `viewingAs` from `localStorage` on init (key: `viewingAsOfficer_${currentUser.id}`)
- Call from `showDashboard()` after the Switch Role init block (~line 472)

#### `selectSwitchedUser(officer)` function
When an officer is selected:
1. Set `window.currentUser.viewingAs = { id: officer.id, name: officer.name, email: officer.email }`
2. Persist to `localStorage` as `viewingAsOfficer_${currentUser.id}` (JSON)
3. Update button visual (add `.active` class, show indicator)
4. Show toast: "Viewing as: {officer.name}"
5. Close popup, re-render dropdown to show active selection
6. **Reload current page view** to re-fetch data with new filter

#### `clearUserSwitch()` function
1. Delete `window.currentUser.viewingAs`
2. Remove from localStorage
3. Reset button visual
4. Show toast: "Switched back to Admin view"
5. Reload current page view

#### Helper: `getViewingAsOfficerName()` 
Returns `window.currentUser?.viewingAs?.name || null` — a clean accessor used by all data-loading pages.

#### Helper: `getViewingAsOfficerId()`
Returns `window.currentUser?.viewingAs?.id || null`

### Step 4: Modify data-loading pages to respect `viewingAs`

Each page that loads data needs to check `window.currentUser.viewingAs` and pass the officer filter when set:

#### 4a. `public/frontend/pages/leads/leadsPage.js`
- In the function that builds the leads fetch URL (~line 346+): if `getViewingAsOfficerName()` is set, append `?assignedTo=OfficerName` to the admin endpoint calls
- Already uses admin endpoints when `role === 'admin'` — just need to add the `assignedTo` query param
- Apply same filter to batch/sheet metadata fetches

#### 4b. `public/frontend/pages/leads/leadManagement.js`
- Similar to leadsPage: when `currentUser.role === 'admin'` and `viewingAs` is set, append `assignedTo` to `/api/crm-leads/admin` calls

#### 4c. Dashboard stats in `public/js/app.js` or dashboard module
- The dashboard stats endpoint returns per-officer breakdowns for admin already
- When `viewingAs` is set, client-side filter the dashboard data to show only that officer's stats, OR pass `?assignedTo=OfficerName` if the endpoint supports it

#### 4d. `public/frontend/pages/calendar/calendarTasks.js`
- When `viewingAs` is set, use `mode=officer&officer=OfficerName` instead of `mode=everyone`

#### 4e. `public/frontend/pages/reports/reportsPage.js`
- Filter report data to the selected officer when `viewingAs` is set

### Step 5: Visual indicator for active impersonation

When viewing as an officer, add a persistent visual cue:
- The user-switch button gets a colored indicator (purple dot or different icon color)
- Optionally show a small banner/chip below the top-bar: "Viewing as: Officer Name [x]" — clicking [x] clears it
